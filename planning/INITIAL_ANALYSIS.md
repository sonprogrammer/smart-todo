# AI Smart Todo Initial Analysis

작성일: 2026-07-18

## 0. 전제

- 현재 저장소에는 구현 코드가 없고, 프로젝트는 초기 기획 단계다.
- 본 문서는 "AI 멀티 에이전트 개발 하네스"와 "Smart Todo"를 함께 만드는 것을 전제로 한 1차 분석이다.
- 구현 기술 스택, 배포 환경, 외부 연동 범위는 아직 확정되지 않았다고 가정한다.
- 따라서 아래 내용은 최종 설계가 아니라, MVP를 시작하기 위한 실행 가능한 기준선이다.

## 1. 프로젝트 요구사항 분석

### 1.1 제품 관점 요구사항

Smart Todo의 핵심 목적은 사용자가 할 일을 단순히 저장하는 것을 넘어서, AI가 작업을 이해하고 정리하며 실행 가능한 형태로 분해해 주는 것이다.

필요한 제품 기능은 다음과 같다.

- 사용자가 todo를 생성, 수정, 완료, 삭제할 수 있어야 한다.
- 사용자가 자연어로 작업을 입력하면 AI가 구조화된 todo로 변환할 수 있어야 한다.
- 하나의 작업을 하위 작업으로 분해하거나 우선순위를 제안할 수 있어야 한다.
- 작업 목록을 상태, 우선순위, 마감일 기준으로 확인할 수 있어야 한다.
- AI가 제안한 결과를 사용자가 확인하고 수용/수정할 수 있어야 한다.

### 1.2 멀티 에이전트 하네스 요구사항

이 프로젝트는 단순 todo 앱이 아니라, 여러 에이전트가 협업하는 실행 하네스를 함께 검증하는 것이 목적이다.

필요한 하네스 요구사항은 다음과 같다.

- 에이전트 역할을 분리할 수 있어야 한다.
- 각 에이전트가 입력, 출력, 책임 범위를 명확하게 가져야 한다.
- 에이전트 간 메시지 전달과 결과 집계를 추적할 수 있어야 한다.
- 하나의 요청이 어떤 에이전트를 거쳐 어떤 결과로 수렴했는지 관찰 가능해야 한다.
- 실패한 에이전트 호출을 재시도하거나 대체 흐름으로 넘길 수 있어야 한다.

### 1.3 비기능 요구사항

- 추적 가능성: AI 응답, 중간 결과, 최종 결과를 기록해야 한다.
- 안정성: 하나의 에이전트 실패가 전체 작업 실패로 직결되지 않아야 한다.
- 확장성: 이후 에이전트 수와 기능이 늘어나도 구조가 무너지지 않아야 한다.
- 테스트 가능성: 에이전트 로직과 todo 도메인 로직을 분리해 단위 테스트가 가능해야 한다.
- 단순성: MVP에서는 과도한 자동화보다 사용자 확인을 우선한다.

## 2. MVP 범위 정의

### 2.1 MVP 목표

MVP의 목표는 다음 두 가지를 동시에 증명하는 것이다.

1. 사용자가 AI 도움으로 todo를 더 빠르고 정확하게 정리할 수 있다.
2. 멀티 에이전트 구조가 실제 제품 흐름에서 관리 가능하고 확장 가능한 방식으로 동작한다.

### 2.2 MVP 포함 범위

- 기본 todo CRUD
- 자연어 입력을 todo 구조로 변환하는 AI 기능
- 하나의 작업에 대한 하위 작업 제안
- 작업 우선순위 제안
- 에이전트 실행 파이프라인의 로그/추적
- AI 결과에 대한 사용자 승인 단계
- 최소한의 단일 사용자 경험

### 2.3 MVP 제외 범위

다음은 MVP에서 제외한다.

- 다중 사용자 협업
- 권한/역할 기반 접근 제어
- 캘린더, 이메일, 메신저 등 외부 서비스 연동
- 음성 입력
- 모바일 네이티브 앱
- 복잡한 워크플로 자동화
- 고급 리포팅/분석 대시보드
- 에이전트 마켓플레이스 또는 플러그인 시스템

### 2.4 MVP 성공 기준

- 사용자가 자연어로 입력한 작업이 구조화된 todo로 변환된다.
- AI 제안 결과를 사용자가 수용하거나 수정할 수 있다.
- 각 요청의 처리 경로와 결과를 추적할 수 있다.
- 최소한의 todo 관리 흐름이 끊김 없이 동작한다.
- 에이전트 분리가 실제 코드 구조에 반영된다.

## 3. 기술적 위험 요소 분석

### 3.1 에이전트 오케스트레이션 복잡도

멀티 에이전트 구조는 분리 자체가 목적이 아니라, 분리한 상태에서도 예측 가능한 결과를 내야 한다.

위험:

- 에이전트가 많아질수록 결과 조합이 불안정해질 수 있다.
- 에이전트 간 책임 경계가 흐려지면 중복 호출과 비효율이 발생한다.
- 한 에이전트의 출력 포맷이 흔들리면 후속 에이전트가 연쇄적으로 실패할 수 있다.

대응:

- 각 에이전트의 I/O 스키마를 고정한다.
- orchestration layer에서 검증과 라우팅을 책임진다.
- 실패 시 fallback 경로를 명시한다.

### 3.2 LLM 출력의 비결정성

위험:

- 동일 입력에 대해 출력 품질이 흔들릴 수 있다.
- 형식이 깨지면 UI나 후속 처리에 영향을 준다.
- 환각된 내용이 todo 데이터에 유입될 수 있다.

대응:

- 구조화 출력 우선으로 설계한다.
- 검증 실패 시 사용자가 확인하도록 되돌린다.
- 생성 결과와 실제 저장 결과를 분리한다.

### 3.3 도메인 모델과 AI 모델의 결합

위험:

- AI 응답 형식이 곧바로 도메인 저장 구조가 되면 변경 비용이 커진다.
- todo 기능 개선과 AI 프롬프트 개선이 서로 엉킬 수 있다.

대응:

- 도메인 모델을 먼저 정의하고, AI는 이를 채우는 보조 계층으로 둔다.
- 변환 계층을 별도로 둬서 모델 변경 영향을 제한한다.

### 3.4 관찰성과 디버깅 어려움

위험:

- 어떤 에이전트가 어떤 판단을 했는지 보이지 않으면 문제 재현이 어렵다.
- 사용자 입장에서 AI가 왜 그런 결과를 냈는지 설명하기 힘들다.

대응:

- 요청 단위 trace id를 둔다.
- 에이전트별 입력/출력/에러를 기록한다.
- 최소한의 실행 로그 뷰를 설계 단계부터 고려한다.

### 3.5 MVP 과설계 위험

위험:

- 처음부터 일반화된 에이전트 플랫폼을 만들면 일정과 복잡도가 급격히 늘어난다.
- 실제 제품 가치보다 하네스 구조에만 시간이 소모될 수 있다.

대응:

- MVP는 todo 도메인 하나에 집중한다.
- 에이전트는 필요한 최소 역할만 둔다.
- 재사용 가능한 추상화는 구현 중 실제 중복이 확인될 때만 도입한다.

## 4. 전체 개발 순서 정의

### Phase 1. 요구사항 고정

- 제품 목적과 MVP 범위를 확정한다.
- todo 도메인에서 반드시 지원할 상태와 속성을 정한다.
- 에이전트 역할과 결과 흐름을 정의한다.
- 저장해야 할 로그와 추적 정보의 최소 범위를 정한다.

### Phase 2. 도메인 모델 설계

- Todo, TaskDraft, AgentRun, AgentResult 같은 핵심 엔티티를 정의한다.
- 사용자 확정 전 draft와 저장된 todo를 구분한다.
- 상태 전이와 검증 규칙을 명확히 한다.

### Phase 3. 하네스 설계

- orchestrator가 에이전트를 호출하는 흐름을 설계한다.
- 각 에이전트의 책임과 입력/출력을 명세한다.
- 실패, 타임아웃, 재시도, fallback 정책을 정의한다.

### Phase 4. AI 처리 파이프라인 구현

- 자연어 입력을 구조화하는 프롬프트 또는 워크플로를 구현한다.
- 작업 분해, 우선순위 제안, 요약 생성 흐름을 구현한다.
- 출력 검증과 사용자 승인 단계를 연결한다.

### Phase 5. 기본 todo 기능 구현

- todo CRUD를 구현한다.
- AI가 생성한 draft를 사용자가 편집하고 저장할 수 있게 한다.
- 상태 변경과 목록 조회를 연결한다.

### Phase 6. 관찰성 및 품질 보강

- 로그, trace, 에러 기록을 정리한다.
- 주요 흐름에 대한 테스트를 추가한다.
- 실패 케이스와 비정상 출력 케이스를 검증한다.

### Phase 7. 확장 준비

- 외부 연동 지점과 향후 에이전트 확장 지점을 분리한다.
- 이후 기능 추가를 위한 인터페이스를 정리한다.

## 5. 초기 결론

이 프로젝트의 핵심은 "todo 앱" 자체보다 "AI 멀티 에이전트 오케스트레이션을 실제 제품 흐름에 안전하게 적용하는 것"이다.

따라서 우선순위는 다음과 같다.

1. todo 도메인의 최소 동작 보장
2. 에이전트 역할 분리와 추적 가능성 확보
3. AI 결과의 구조화와 사용자 승인
4. 이후 기능 확장 가능한 경계 설계

이 기준으로 다음 단계에서는 기술 스택과 디렉터리 구조, 그리고 구체적인 구현 계획을 수립하는 것이 적절하다.
