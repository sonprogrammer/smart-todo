# AI Smart Todo Development Plan

작성일: 2026-07-18

## 0. 계획 기준

이 계획은 [INITIAL_ANALYSIS.md](./INITIAL_ANALYSIS.md)의 분석 결과를 기준으로 작성한다.

핵심 우선순위는 다음과 같다.

1. todo 도메인의 최소 기능을 안정적으로 제공한다.
2. AI 멀티 에이전트 하네스의 역할 분리와 추적 가능성을 확보한다.
3. AI 결과를 사용자 승인 흐름과 연결해 잘못된 자동 저장을 막는다.
4. 이후 기능 확장을 막지 않는 경계와 구조를 만든다.

## 1. Planning 단계

### 목표

- 프로젝트 요구사항을 구현 가능한 수준으로 고정한다.
- MVP 범위를 확정하고, MVP 밖의 기능을 명확히 제외한다.
- todo 도메인, AI 처리, 오케스트레이션, 로그/추적의 경계를 정의한다.
- 이후 단계에서 사용할 공통 용어와 성공 기준을 확정한다.

### 산출물

- 프로젝트 요구사항 정리
- MVP 범위 정의
- 기술 리스크 목록
- 개발 우선순위 및 단계별 순서
- 용어와 핵심 엔티티 초안

### 완료 조건

- MVP에 포함/제외할 기능이 문서로 명확히 구분되어 있다.
- Smart Todo와 멀티 에이전트 하네스의 목적이 한 문장 수준으로 설명된다.
- 도메인 모델의 핵심 엔티티 후보가 정리되어 있다.
- 기술적 위험과 대응 방향이 식별되어 있다.
- 다음 단계인 Architecture 설계를 시작할 수 있을 만큼 범위가 충분히 좁혀져 있다.

## 2. Architecture 단계

### 목표

- 전체 시스템을 도메인, 오케스트레이터, AI 에이전트, 저장소, UI 계층으로 분리한다.
- 각 컴포넌트의 책임과 인터페이스를 정의한다.
- AI 출력이 도메인 저장 구조에 직접 침투하지 않도록 경계를 설계한다.
- 실패, 재시도, 검증, 승인 흐름을 포함한 실행 모델을 정한다.

### 설계 범위

- todo 도메인 모델
- task draft와 확정 todo의 분리
- agent run / result 추적 구조
- orchestrator와 개별 agent의 I/O 계약
- 검증과 fallback 전략
- 로그와 trace 구조

### 완료 조건

- 시스템 구성도와 데이터 흐름이 설명 가능하다.
- 각 agent의 책임이 겹치지 않도록 구분되어 있다.
- 입력, 중간 결과, 최종 결과를 추적할 수 있는 구조가 정의되어 있다.
- AI 생성 결과와 영속 저장 모델의 경계가 분리되어 있다.
- 구현 단계에서 사용할 인터페이스 초안이 충분히 구체적이다.

## 3. Frontend 단계

### 목표

- 사용자가 todo를 생성하고 AI 제안을 검토하는 핵심 화면을 구현한다.
- 자연어 입력, AI 제안 미리보기, 사용자 승인, todo 목록 확인 흐름을 연결한다.
- 과도한 장식보다 작업 중심의 단순하고 명확한 UI를 제공한다.

### 핵심 화면

- Todo 목록 화면
- Todo 작성/편집 화면
- 자연어 입력 및 AI 제안 결과 화면
- Draft 검토 및 승인 화면
- 실행 기록 또는 추적 확인 화면

### 완료 조건

- 사용자가 todo를 생성, 수정, 완료, 삭제할 수 있다.
- 자연어 입력 후 AI 결과를 확인하고 승인 또는 수정할 수 있다.
- 리스트와 상세 화면 간 이동이 끊기지 않는다.
- AI 결과가 실제 저장 전 draft 상태로 구분되어 보인다.
- 기본적인 오류 및 로딩 상태가 표현된다.

## 4. Backend 단계

### 목표

- todo CRUD와 AI 파이프라인을 연결하는 서버를 구현한다.
- 오케스트레이터가 agent를 호출하고 결과를 집계하는 흐름을 구현한다.
- 저장소에 todo, draft, run log, result를 일관되게 저장한다.
- 입력 검증과 출력 검증을 통해 AI 비결정성의 영향을 줄인다.

### 구현 범위

- todo 도메인 API
- draft 생성 및 승인 API
- orchestrator 실행 API
- agent별 실행 로직
- 결과 검증과 에러 처리
- trace / log 저장 구조

### 완료 조건

- todo CRUD가 API 수준에서 안정적으로 동작한다.
- AI 요청이 draft를 생성하고, 사용자가 승인해야 실제 todo로 저장된다.
- 에이전트 실행 기록을 조회할 수 있다.
- 실패 케이스에 대해 명확한 에러 응답과 fallback이 존재한다.
- 도메인 로직과 AI 처리 로직이 분리되어 테스트 가능한 상태다.

## 5. QA 단계

### 목표

- 핵심 사용자 흐름이 안정적으로 동작하는지 검증한다.
- AI 출력 오류, 검증 실패, 부분 실패, 재시도 같은 비정상 케이스를 확인한다.
- 도메인 규칙과 오케스트레이션 규칙이 예상대로 유지되는지 점검한다.

### 검증 범위

- todo CRUD 기능
- 자연어 입력에서 draft 생성까지의 흐름
- draft 승인 후 저장 흐름
- agent 실행 로그와 trace 기록
- 실패 및 비정상 출력 처리
- 핵심 데이터 무결성 검증

### 완료 조건

- 주요 사용자 플로우에 대해 성공/실패 케이스가 검증되어 있다.
- 구조화 출력이 깨졌을 때의 처리 방식이 확인되어 있다.
- 에이전트 한 개가 실패해도 전체 시스템이 무너지지 않는다.
- 회귀를 막기 위한 테스트가 핵심 경로에 존재한다.
- MVP 범위 안의 기능에 대해 릴리스 가능한 품질 수준에 도달했다.

## 6. Review 단계

### 목표

- 구현 결과가 초기 분석과 계획을 충족하는지 검토한다.
- 과설계, 중복 책임, 불명확한 인터페이스를 제거한다.
- 다음 확장 단계로 넘어가기 전에 기술 부채와 품질 리스크를 정리한다.

### 검토 항목

- 요구사항과 실제 구현의 일치 여부
- MVP 범위 준수 여부
- 도메인 모델의 안정성
- agent 분리와 오케스트레이션 명확성
- 로그/추적의 충분성
- 테스트 커버리지와 비정상 케이스 대응

### 완료 조건

- 계획된 MVP 목표가 실제 결과로 검증된다.
- 불필요한 기능이 포함되어 있지 않다.
- 핵심 흐름의 문제점이 식별되고 정리되어 있다.
- 다음 반복에서 개선할 항목이 우선순위와 함께 정리되어 있다.
- 사용자 관점에서 최소한의 Smart Todo 경험이 완결된다.

## 7. 전체 진행 순서

1. Planning 단계에서 범위와 성공 기준을 고정한다.
2. Architecture 단계에서 시스템 경계와 데이터 흐름을 정의한다.
3. Frontend 단계에서 사용자 입력과 AI 검토 경험을 구현한다.
4. Backend 단계에서 도메인, 오케스트레이션, 저장, 추적을 연결한다.
5. QA 단계에서 핵심 플로우와 실패 케이스를 검증한다.
6. Review 단계에서 결과를 정리하고 다음 반복을 위한 개선점을 도출한다.

## 8. 최종 원칙

- MVP를 넘는 기능은 다음 반복으로 미룬다.
- AI는 자동 결정보다 보조와 제안에 초점을 둔다.
- 도메인 모델은 AI 프롬프트보다 안정적으로 유지한다.
- 추적 가능성은 부가 기능이 아니라 필수 기능으로 취급한다.
- 구현이 진행되더라도 이 문서의 단계 정의가 실행 기준이 된다.
